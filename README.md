## optimze4：双库解耦架构与防御性工程重构 (Storage & Engineering)

本分支聚焦于解决由于复杂非结构化数据和恶意并发带来的系统物理崩溃，完成了从单机单库到微服务彻底解耦的架构涅槃。

*   **痛点 1：向量库物理极限爆仓 (The 65535 Limit)**
    *   **问题**：处理含无标题长表格的财报时，切分器切出了长达 8 万字的“超级父块”，直接撑爆了 Milvus 单个 VARCHAR 字段 65535 字节的物理硬限制，导致入库流水线崩溃。同时，用昂贵的向量库内存去存储十万字的低频长文本，导致了极大的资源浪费。
    *   **方案**：实施**“计算与存储彻底解耦”**的微服务战略。让 Milvus 退化为纯粹的“计算雷达”，仅负责存储 500 字的精细子块与稠密向量进行高速数学计算。同时引入 PostgreSQL 关系型数据库作为持久化“仓库”，利用其 Text 字段无长度限制的特性专职存储超级父块原文。两者通过 `parent_id` 实现 O(1) 映射，消灭了物理截断报错，实现了长表数据 0 丢失。

*   **痛点 2：内存 OOM 与重复数据带来的生态污染**
    *   **问题**：Docling 解析长图文瞬间索要几 GB 连续内存极易引发 OOM；更致命的是，同一份文件的重复上传会导致向量检索的 Top-K 召回高度同质化（信息多样性降为 0），并导致 BM25 词频失真，白白消耗高昂的算力和 API 资金。
    *   **方案**：
        *   **防 OOM 兜底**：引入流式解析机制与批处理（Batching）调度，将海量切块按 `batch_size=20` 平滑请求大模型，根除网络阻断与内存溢出。
        *   **去重防弹盾（安检左移）**：弃用容易状态撕裂的异步查重，引入基于 MD5 的哈希指纹去重机制。采用分块流式计算（Streaming Hash）提取指纹，并将校验逻辑**前置（Shift-Left）**到 HTTP 请求的主线程中。命中重复文件秒级阻断并抛出 `409 Conflict`，杜绝了重复数据对向量空间的污染，提升了系统的 ROI 与前端交互体验。
